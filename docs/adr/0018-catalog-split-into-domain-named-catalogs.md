# The single `catalog` splits into eight domain-named catalogs

_Decided 2026-07-23._

The root `package.json`'s `workspaces.catalog` — a flat, unnamed list of 23 pinned dependency versions — is replaced with `workspaces.catalogs`, Bun's named-catalog feature. There is no bare default catalog anymore: every cataloged dependency lives under one of `auth`, `database`, `react`, `rpc`, `testing`, `env`, `observability`, or `core`, referenced from consuming `package.json` files as `catalog:<name>` instead of the old bare `catalog:`. This is a pure reorganization — no dependency was added, removed, or re-versioned; only which named catalog owns it changed.

Two placements aren't obvious from the dependency name alone and are worth recording. `zod` sits in `rpc` alongside `@orpc/server` and `@tanstack/react-query` rather than in `core`, because its dominant use in this repo is validating oRPC contracts and schemas, not general-purpose validation — grouping it with the RPC layer keeps that version tied to the layer it actually serves. `evlog` sits in its own `observability` catalog rather than folded into `core`; it's the only entry today, but the catalog is named for the category (logging, and eventually tracing/metrics tools like Sentry or PostHog) rather than for the single package currently in it.

`testing` (`vitest`, `@vitest/ui`, `@vitest/coverage-istanbul`, `jsdom`) stays a separate catalog from `core` (`typescript`, `@types/bun`, `better-result`) even though both are referenced from nearly every package in the monorepo. The distinction is what the dependency is *for*, not how widely it's used: `core` holds primitives that ship in production code paths (the compiler, ambient Bun types, the error-as-values helper), while `testing` is exclusively dev-time tooling. Collapsing the two would turn `core` into a catch-all for "anything widely referenced," which defeats the purpose of naming catalogs by domain.

`.syncpackrc.json`'s single `policy: "catalog"` version group is split the same way, into eight domain-labeled groups (syncpack has first-class support for named catalogs, so this required no change to how the policy itself is enforced). While splitting it, `@vitest/ui` and `@vitest/coverage-istanbul` were added to the `testing` group — both were already catalog entries consumed via `catalog:` but had been missing from syncpack's enforcement list.
